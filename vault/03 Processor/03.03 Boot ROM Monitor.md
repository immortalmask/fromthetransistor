# Boot ROM Monitor

## Why this block exists

A working CPU is difficult to use if every program must be compiled into the simulator. A boot monitor creates a small trusted recovery path: receive an image, validate it, place it in executable memory, report errors, and transfer control. The protocol also forces byte order, MMIO side effects, memory regions, and trust boundaries to agree.

## Lecture

The FTT-16 lab intentionally has separate instruction words and data bytes and no devices. Do not pretend its current `store` instruction can rewrite instruction memory. This module makes the necessary extension explicit: a simulated **boot-load device** exposes fixed MMIO registers in the upper data-address range. Writes stream validated instruction bytes into a bounded instruction-RAM window; the device, not an ordinary data-memory cast, owns that write port.

A coherent teaching map is:

```text
instruction words 0..31    immutable boot ROM
instruction words 32..127  loadable program RAM
data 0x00..0xef             ordinary data RAM
data 0xf0..0xff             UART and boot-load MMIO
```

The exact register table belongs in the artifact. It needs RX ready/data, TX ready/data, boot data, boot status, and commit/abort behavior. Reading or writing each register must have one documented side effect.

Use a bounded protocol such as:

```text
magic[2] version[1] word_count[1] payload[word_count*2] checksum[1]
```

Choose and document payload byte order and checksum rule. `word_count` must be 1..96 for the proposed program window. Validate magic/version/count before payload writes where possible; stage incoming words in simulator-side boot memory and expose them to instruction RAM only after the full checksum succeeds. On error, abort the staged image, send a stable status byte, and return to sync search.

This is a model of a ROM monitor and DMA-like loader, not a claim that the eight-instruction lab ISA alone implements every buffer and checksum operation. An assembly polling loop can drive the fixed MMIO stream; the simulator-side device safely owns staging and commit.

## Workbook

Design the MMIO table with columns `address`, `name`, `read result`, `read side effect`, `write effect`, and `reset value`. Predict the status sequence for one received byte, one consumed byte, and one transmitted reply.

For a two-word payload `122a 0000`, encode the payload bytes in the byte order established by the assembler module. Construct a complete frame under your protocol and calculate the checksum by hand.

Trace three transfers:

1. valid two-word image;
2. valid header but one payload byte missing;
3. bad checksum followed immediately by a valid image.

Use rows `input offset`, `parser state`, `remaining`, `staged words`, `checksum`, `reply`, and `committed?`. The bad image must never partially replace executable RAM, and the next magic must be discoverable.

Sketch the assembly polling shape using only existing FTT-16 operations: `load` fixed status, `jz` back to poll, `load` fixed data, `store` to fixed boot-data register, and branch. Identify which complex validation is delegated to the boot-load device and why.

## Problem set

1. Why can ordinary FTT-16 data `store` not load the existing instruction array?
2. If ROM occupies 0..31 and RAM 32..127, what is the largest accepted word count? Check the zero case too.
3. A length field is trusted before range checking. Describe two state-corruption outcomes.
4. Distinguish checksum, authentication, and range validation. Which attacks or accidents does each address?
5. After receiving half a frame, input stops forever. What execution bound or timeout keeps the simulation useful?
6. Why is commit-after-validation safer than writing instruction RAM as each byte arrives?

## Homework

Build a host-side/simulator boot monitor model plus a small assembly polling-loop design:

1. Freeze the instruction/data/MMIO map and byte-level frame format.
2. Implement a parser with explicit states and bounded counts.
3. Stage complete 16-bit words and commit only after valid magic, version, count, and checksum.
4. Add stable replies for success, bad header, bad length, bad checksum, timeout, and abort.
5. Feed the committed words to the existing CPU emulator, set PC to word 32, and show a loaded program halting with its expected final state.
6. Inject truncation at every frame byte and prove no partial image becomes executable.
7. Demonstrate bad frame followed by valid frame to prove resynchronization.

The artifact is the protocol/register specification, parser model, FTT-16 polling-loop sketch or implementation, fault matrix, and complete host-send → validate → commit → execute transcript.

## Reverse-engineering lens

Boot protocols are often recovered from captures and status behavior. Identify fixed magic, version, length correlations, payload byte order, and integrity bytes across several images. Corrupt one field at a time in simulation and record reply/state changes. Never probe undocumented physical boot pins or firmware-update services without authorization.

## Exit criteria

- [ ] The memory/MMIO map makes instruction loading an explicit device operation.
- [ ] Every header, range, byte-order, and checksum rule is written and tested.
- [ ] Truncated or corrupt images never partially commit executable state.
- [ ] A valid image loads into words 32..127 and executes under the CPU emulator.
- [ ] The monitor resynchronizes after an invalid frame and retains a recovery path.
