# Shell Build and Debug Loop

## Why this block exists

Layered systems amplify weak debugging habits. If compilation, exit status, standard streams, test selection, and crash inspection are vague, a one-line C fault becomes an imagined processor bug. This block makes the local loop short and evidence-producing before the machine has many layers.

## Lecture

A useful loop is **inspect → predict → build → run one case → capture → explain**. The shell connects programs; it does not make their failures disappear. Learn to distinguish:

- compilation failure from link failure;
- nonzero process status from incorrect printed output;
- standard output from diagnostic output;
- a deterministic test from a timing accident;
- a symptom frame from the frame where bad state was created.

Compile learner C with a strict host profile while learning:

```sh
cc -std=c17 -Wall -Wextra -Wpedantic -Wconversion -Werror -g \
  -fsanitize=address,undefined program.c -o program
```

Warnings are design feedback, not decorations. Do not silence one with a cast until you can state the conversion's range and intent. Sanitizers are runtime oracles for some classes of invalid behavior, not proofs of correctness.

Shell quoting is part of correctness. A filename, byte string, or generated argument must not be reinterpreted as syntax. Prefer explicit commands and small scripts over dense chains whose failing segment is unclear.

In a debugger, begin with the fault, inspect the backtrace and arguments, then move toward the earliest corrupted value. Record the input and exact build; otherwise the crash is not a fixture.

## Workbook

Before compiling, predict which phase reports each defect:

| Defect | Compile | Link | Run | Test oracle |
|---|---:|---:|---:|---:|
| missing semicolon | ? | ? | ? | ? |
| declaration exists but definition is absent | ? | ? | ? | ? |
| write one byte past a local array | ? | ? | ? | ? |
| prints `41` when expected output is `42` | ? | ? | ? | ? |

Create a temporary two-file program. Compile each source to an object, inspect the filenames and sizes, then link them. Run it and record `$?`. Next, introduce one undefined symbol and identify the phase that rejects it.

Create a one-element out-of-bounds access in temporary code. Predict whether compilation catches it. Run the sanitized build, capture the first diagnostic frame, and identify the object plus invalid offset. Restore the safe version.

Finally, compare these debugging statements:

- “It crashed in `print_result`.”
- “Input length 17 wrote index 16 into a 16-byte object in `decode`; `print_result` later read the damaged terminator.”

Explain why only the second describes causality.

## Problem set

1. A build command returns status 0 but a later command fails. What should a useful script report?
2. Why can optimized and sanitized builds expose different symptoms without either being the “real” machine behavior?
3. A warning says comparison mixes signed and unsigned values. Give a negative input that changes the comparison and repair the data model rather than adding a cast.
4. Explain why deleting all generated files is a poor first response to a deterministic unit-test failure.
5. A test passes alone and fails after another test. Name two ownership/state hypotheses and one trace for each.

## Homework

Write a small command-line C program split across at least two source files. It should parse one integer, reject trailing text and out-of-range values, compute a simple result, and return distinct nonzero statuses for usage versus invalid data.

Deliver:

1. A strict warning-clean build command.
2. Tests for empty input, valid boundaries, just-outside boundaries, and trailing junk.
3. One deliberately introduced defect and a captured sanitizer or debugger trace.
4. A repair plus an explanation naming the violated invariant.
5. A short build map showing source → object → executable.

Do all destructive experiments in temporary learner files. Never modify system binaries, course solution trees, or files outside the repository/work area.

## Reverse-engineering lens

A crash address is only an observation. Use symbols, disassembly, stack frames, register values, and object bounds to turn it into a causal story. Practice separating “where execution stopped” from “where invalid state originated”; this distinction returns in page faults and packet parser failures.

- [ ] You captured and explained one reproducible failure before repairing it.
- [ ] You can inspect process status, standard output, and diagnostics separately.
