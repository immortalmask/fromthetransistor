# CPU Emulator

## Why this block exists

The CPU gives instruction bits causal meaning. A software emulator exposes the complete architectural state after every step, making it the best first implementation and later a differential oracle for RTL. Starting with a pipeline would add timing hazards before the basic instruction semantics are trustworthy.

## Lecture

The FTT-16 lab machine has:

- up to 128 supplied 16-bit instruction words;
- a program counter measured in **instruction-word indexes**;
- sixteen 8-bit registers initialized to zero;
- 256 zero-initialized data bytes;
- the eight encodings from the assembler module;
- a hard execution budget of 256 instructions.

One step follows **fetch → validate/decode → compute → commit**. Keep `next_pc` explicit. Most instructions default to `pc + 1`; `jmp` replaces it; a taken `jz` adds the sign-extended offset to `pc + 1`. `halt` counts as an executed step and leaves PC at the halt instruction.

Architectural changes must be precise. Validate an encoding and every target before committing any register, memory, or PC change. `add` wraps modulo 256. `load` and `store` access data space, not the instruction array. Runtime errors—invalid encoding, out-of-range PC, or exhausted budget—return differently from malformed command input.

The 256-step budget converts an accidental infinite loop into deterministic evidence. It is not a performance model or an operating-system timer.

Use a trace row shaped like:

```text
step pc word decoded read-values write-effect next-pc
```

Final-state comparison catches many bugs; first-divergence traces localize them. An optional RTL core should be single-cycle first and compared against the emulator at architectural commit points.

## Workbook

Trace this program before running it:

```text
index  word  assembly
0      11fe  ldi r1, 254
1      1205  ldi r2, 5
2      2120  add r1, r2
3      5110  store r1, 0x10
4      4310  load r3, 0x10
5      0000  halt
```

Write every changed register, memory byte, PC, and step count. Explain why `r1` becomes `03`.

Trace a taken branch:

```text
1100 7101 122a 1207 0000
```

At `jz`, compute both fall-through and taken next PCs before selecting. Which `ldi` is skipped?

Predict the exact class of result for programs `6000`, `6001`, `8000`, and an empty command line. Separate input error, runtime error, and successful halt.

Start the CPU lab:

```sh
python3 ftt start 03.02
python3 ftt check 03.02
```

Feed it four-hex-digit words produced by the assembler lab. Add local cases for not-taken branch, backward branch, jump out of range, reserved low nibble, memory zero, and memory nonzero ordering.

## Problem set

1. Why is PC 3, not 4, in a four-word program whose final word is `halt`?
2. A taken `jz` at PC 0 has offset `-2`. Show the mathematical target and explain why validation must occur before converting to an unsigned index.
3. Write the full state transition for `xor r4, r4`.
4. An invalid `add` encoding also names valid registers. Which check wins, and which state may change?
5. Explain how shared decode code between assembler and emulator can hide a common bug. Propose an independent oracle.
6. A program loops for 256 steps and would halt on step 257. State the required result under the declared budget.

## Homework

Complete the automated emulator in stages:

1. Parse all words and establish zeroed state before execution.
2. Implement halt and `ldi`, then ALU, memory, and control flow, adding tests at each stage.
3. Reject reserved encodings before reading operand fields as effects.
4. Enforce PC bounds and the 256-step budget with distinct runtime diagnostics.
5. Add an optional trace mode in learner-only code whose output records every committed transition without changing required output.
6. Run assembler-produced programs for wrapping arithmetic, store/load, branch taken/not taken, and a deliberate loop.
7. As a stretch, implement a single-cycle RTL state transition and compare commit traces against the C emulator.

The artifact is the passing emulator, at least five hand traces, and a first-divergence report from one deliberately injected decode or PC bug.

## Reverse-engineering lens

Given only final registers, multiple instruction sequences may fit. Add a commit trace to constrain the history. Conversely, when a captured trace begins mid-program, use known instruction semantics to work backward from writes and control-flow targets. Watch the distinction between an address expressed in instruction words and data bytes; unit confusion is a common reversing error.
