# 04.01 C Compiler

## Why this block exists

A compiler is the first component in the course whose input looks friendly while its failures appear far away as wrong machine state. Building a deliberately small compiler makes the source-to-instruction contract visible and prepares you to debug generated code rather than treating a host compiler as an oracle.

## Lecture

### Mental model

Treat compilation as a sequence of evidence-preserving transformations: bytes become tokens, tokens become an abstract syntax tree, names become resolved objects, expressions gain types, control flow becomes explicit, and target-independent operations become assembly. Every stage accepts one representation and either produces a valid next representation or a location-aware diagnostic. Do not let parsing, type checking, and code generation silently repair one another.

The course language is **C0**, not all of ISO C: integer and pointer types, local variables, functions, calls, `if`, `while`, arrays, dereference, address-of, and explicit returns. Freeze its grammar, integer widths, evaluation rules, calling convention, and rejected constructs before implementation.

### Invariants

- Every token and AST node retains a source span.
- Each identifier use resolves to exactly one declaration before code generation.
- Stack slots are aligned, non-overlapping, and addressed from one documented frame base.
- Every reachable function path either returns a value of the declared type or is diagnosed.
- Generated stack depth returns to its entry value at every control-flow join.

### Safety and limits

Reject unsupported C instead of guessing. Avoid compiling and running unknown source outside the course emulator. C0 excludes preprocessing, floating point, variadic calls, concurrency, and undefined host-dependent behavior; those omissions are a specification boundary, not missing error handling.

## Workbook

Given `int f(int x) { int y = x + 3; if (y < 8) return y * 2; return y - 1; }`, first write the token stream and AST. Then predict the symbol table, stack offsets, branch labels, and stack depth after each emitted instruction. Compile it and annotate the first stage where your prediction differs. Repeat after replacing `y` with an undeclared name and require the diagnostic to point to that use.

Trace the expression `*p++ + a[i]` under the documented C0 precedence rules. State whether it is accepted, which subexpression changes `p`, and the order assumptions your compiler must not invent.

## Problem set

1. Write an unambiguous grammar for calls, indexing, unary operators, and binary precedence; show the parse tree for `a + b * c == d`.
2. Explain why lexing `x-1` as three tokens is preferable to treating `-1` as one token in this grammar.
3. Design diagnostics for duplicate declarations, an unknown identifier, wrong call arity, and a missing return. Include the source span each uses.
4. Draw a stack frame for two parameters, three locals of different widths, saved return state, and one outgoing call. Calculate all padding.
5. Lower a `while` containing an `if` and an early return to labeled basic blocks; identify every join and stack-depth assertion.
6. Find a counterexample showing why evaluating both sides of `&&` is observably wrong.

## Homework

Build the compiler in stages: (1) tokenizer with spans and malformed-token tests; (2) recursive-descent parser with printed AST snapshots; (3) scoped name resolution and a small type checker; (4) assembly generation for expressions and calls; (5) statements and control flow; (6) a driver that emits assembly or deterministic diagnostics. Keep golden fixtures for every intermediate representation.

Run `python3 ftt start lab-04-compiler`, work only in the generated directory, then use `python3 ftt check lab-04-compiler`.

Acceptance rubric: 35% valid programs compile and execute correctly, 20% invalid programs fail at the correct stage with useful spans, 15% calling convention and stack invariants, 15% learner-written boundary tests, 10% readable stage separation, and 5% a short limitations note. A crash, silent acceptance of unsupported syntax, or target-dependent output is not a pass.

## Reverse-engineering lens

Compile several tiny functions with the host compiler at low optimization, disassemble them, and infer parameter locations, prologue shape, and return convention. Then compare that evidence with C0 output. The goal is not byte-for-byte imitation; it is explaining each discrepancy from two documented ABIs.

## Exit criteria

- [ ] You can trace one source expression through tokens, AST, types, and instructions.
- [ ] Valid C0 programs have deterministic assembly and invalid programs have source-located errors.
- [ ] Calls, returns, branches, and stack slots obey written invariants.
- [ ] You can diagnose a wrong result by naming the first incorrect representation.
