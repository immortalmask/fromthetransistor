# 06.03 Dynamic Linking

## Why this block exists

Static linking duplicates common code in every process. A small dynamic loader shows that shared libraries are ordinary mapped objects plus a runtime algorithm for dependencies, symbols, relocations, and final permissions.

## Lecture

### Mental model

The kernel maps the interpreter and initial program; the user-space loader validates dependency metadata, maps load segments at chosen bases, builds a global symbol scope, applies supported dynamic relocations, establishes entry state, and transfers control. A symbol address is object load bias plus symbol value. Position-independent code reaches mutable addresses through explicit indirection rather than embedding one fixed load address.

Keep discovery, mapping, symbol lookup, relocation, and protection finalization as distinct phases. Track an object graph so dependencies load once and cycles terminate. Apply relocations only to writable validated targets, then remove write access from executable/read-only regions.

### Invariants

- Each dependency identity maps once per process even if reached by multiple paths.
- Every relocation target lies inside a mapped writable region during relocation.
- Symbol lookup follows one documented scope and reports unresolved/ambiguous cases deterministically.
- Relocation arithmetic fits its field and uses the correct defining object's base.
- Writable-and-executable mappings do not remain after finalization.

### Safety and limits

Load only course-produced objects from an allowlisted virtual filesystem. Reject absolute paths, traversal, malformed ranges, and unsupported relocation types. The core omits arbitrary host shared objects, TLS, lazy binding, symbol versioning, auditing, and set-user-ID security; retain a static-link path for the browser.

## Workbook

Map a main object and two libraries at three supplied bases. Resolve six symbols under the documented breadth-first scope, apply relative, absolute, and jump-slot-style teaching relocations, and predict final page permissions. Repeat with a dependency cycle and a duplicate symbol.

Draw the same code loaded at two different bases. Mark values that change, values that remain object-relative, and the indirection required to avoid modifying code bytes.

## Problem set

1. Calculate relocated values for `B + A`, `S + A`, and `S + A - P` at two load bases.
2. Resolve symbols across main, library A, and library B under two lookup orders; show one behavior difference.
3. Design cycle-safe dependency loading that still preserves deterministic initialization order.
4. Explain why text relocations conflict with sharing and write-xor-execute policy.
5. Validate a segment whose file range is valid but whose memory range wraps the virtual address space.
6. Specify rollback if the third dependency fails after two objects were mapped and partly relocated.

## Homework

Implement: (1) bounded dynamic-object parser; (2) dependency graph and deterministic mapping; (3) global symbol scope; (4) relative and symbol relocations; (5) final memory protections and entry handoff; (6) module reuse across two simulated processes; (7) failure rollback and trace output. Compare results at multiple randomized load bases.

Acceptance rubric: 20% parsing/mapping, 25% symbol resolution, 25% relocations/rebasing, 10% dependency cycles and reuse, 10% permissions/rollback, and 10% tests. Unsupported objects must fail before partial control transfer.

## Reverse-engineering lens

Given a memory map, dynamic table, and relocated pointer values but no loader log, reconstruct object bases and symbol providers. Use one unresolved call site to determine whether the defect is lookup order, missing dependency, or wrong relocation arithmetic.

## Exit criteria

- [ ] You can calculate supported dynamic relocations at arbitrary load bases.
- [ ] Dependencies, cycles, and symbol scope are deterministic.
- [ ] Relocation writes are bounded and final permissions remove writable code.
- [ ] Static linking remains a supported fallback for the final browser.
