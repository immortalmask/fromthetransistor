# Structs Memory and Ownership

## Why this block exists

C lets the course define its own process, packet, page, file, and parser objects, but it does not define who frees them or whether a half-built object is usable. Kernels fail when those lifecycle rules live only in the author's head. This block turns them into API contracts and invariants.

## Lecture

A struct groups fields under one type. The implementation may insert padding for alignment, so `sizeof(struct)` need not equal the sum of field sizes. Padding bytes may be indeterminate and host layout is not a wire format. Encode external formats field-by-field.

Use one of three ownership words at interfaces:

- **borrowed:** usable only while the owner's documented lifetime remains valid;
- **owned:** the receiver must eventually release it with the matching operation;
- **transferred:** successful completion moves the release responsibility; define what failure does.

Initialization establishes an invariant. Destruction should accept every successfully initialized state and release each owned resource exactly once. For a multi-step constructor, either build into a temporary and commit only when complete, or maintain a cleanup path that knows which steps succeeded.

Heap allocation can fail. Compute `count * sizeof(element)` only after checking multiplication. A zero-length collection still needs a clear valid representation. Reallocation may move storage; do not retain interior pointers across it.

Global and static state makes ownership and test isolation harder. Use it only when the machine contract requires one shared instance, and provide explicit reset behavior for tests.

## Workbook

For this type, predict likely offsets, then verify them with `offsetof` rather than assuming:

```c
struct item {
    uint8_t kind;
    uint32_t length;
    uint8_t flags;
};
```

Reorder fields and compare size. Explain why neither layout may be copied directly into an on-disk record.

Trace a constructor that allocates a struct, then an array, then copies a name. Make a row for failure after each step and list the resources that must be released. State the caller-visible result after failure.

Draw ownership arrows for an object containing an owned array and a borrowed label. What becomes invalid if the label owner is destroyed first?

## Problem set

1. Why can comparing two structs with `memcmp` produce a false difference even when all named fields match?
2. Write the guard needed before allocating `count * sizeof(struct item)`.
3. A successful `collection_add` takes ownership of `item`; failure leaves ownership with the caller. Show both cleanup paths.
4. Explain why `free(NULL)` being valid does not make double-free safe.
5. A pointer into a growable array is saved, then the array reallocates. Diagnose the later failure.

## Homework

Implement an owned collection of named byte blobs:

1. Define a struct with explicit owned fields and length/capacity values.
2. Provide `init`, `append_copy`, `find`, and `destroy` operations.
3. Make append transactional: allocation failure leaves the original collection valid and unchanged.
4. Reject length arithmetic overflow before allocation.
5. Add a test allocator or controlled failure hook so each allocation step can be forced to fail.

Your invariant should state: initialized shape, bounds between length and capacity, ownership of every live element, and allowed zero-length representation. Deliver an invariant table, failure-path trace, and sanitizer-clean tests.

## Reverse-engineering lens

Struct layouts are often inferred from repeated offsets, but one function rarely proves a field's type. Track reads, writes, widths, allocation sizes, and cleanup calls across functions. Destructor behavior is especially informative: it reveals which fields are pointers, which are owned, and which subobjects may be optional.

## Exit criteria

- [ ] You explain observed padding without treating host layout as a protocol.
- [ ] Every pointer field in the artifact has a stated owner and lifetime.
- [ ] Forced failure at every allocation site leaves no leak or half-valid object.
- [ ] Size arithmetic is checked before allocation.
- [ ] Initialization and destruction preserve the written invariant.
