# 04.03 Freestanding libc and Allocator

## Why this block exists

Generated C cannot become a system without startup code, byte and string routines, formatting, and a memory-allocation policy. Rebuilding this narrow runtime exposes object lifetimes and ownership rules that hosted environments conceal.

## Lecture

### Mental model

Freestanding code begins at an ABI entry point, establishes stack/global state, and calls `main` without assuming an operating system. Memory primitives operate on byte ranges with explicit lengths. An allocator manages one arena by partitioning it into aligned allocated blocks and free blocks; metadata is part of that partition, not magical bookkeeping outside it.

Implement a bump allocator first to establish alignment and exhaustion semantics. Then use an address-ordered free list with boundary tags or explicit sizes. Allocation splits only when the remainder can hold valid metadata plus minimum payload. Freeing validates ownership, inserts once, and coalesces adjacent free blocks.

### Invariants

- Every managed byte belongs to exactly one block or documented allocator metadata.
- Payload pointers have the promised alignment and fall inside the arena.
- Free-list nodes are in range, non-overlapping, ordered, and cycle-free.
- Adjacent free blocks are coalesced after `free`.
- `memmove` handles overlap; `memcpy` either documents non-overlap or rejects misuse in tests.

### Safety and limits

Use checked addition for sizes and alignment rounding. Never dereference a candidate block header before proving it is in the arena. Core `printf` supports a narrow set of conversions and bounded output; it is not a full locale-aware libc. Double-free and foreign-pointer behavior must be detected in the teaching allocator rather than becoming silent corruption.

## Workbook

For a 128-byte arena with 8-byte alignment and 8-byte headers, predict the exact block list after allocations of 13, 8, and 27 bytes; then free the middle, free the first, and allocate 18. Record requested size, rounded payload, split remainder, and free-list links at every step. Compare your trace with a heap-dump function.

Trace `memmove(buf + 2, buf, 6)` over `ABCDEFGH`, one byte at a time. Predict the wrong result from a forward-only loop and state the range test that selects direction.

## Problem set

1. Prove why `size + alignment - 1` can overflow and write a checked rounding formula.
2. Choose a minimum split size for a given header and alignment; show a request that would create an unusable fragment without it.
3. Diagnose a free list containing two overlapping nodes and a cycle. Which invariant should fail first and why?
4. Specify `strlen`, `strnlen`, `memcpy`, `memmove`, and `memcmp` for zero length and null pointers without importing host assumptions.
5. Design a bounded integer formatter that handles zero, the most negative signed value, insufficient output space, and bases 10 and 16.
6. Compare first-fit and best-fit using one allocation trace; distinguish measured fragmentation from intuition.

## Homework

Stage the runtime as: (1) byte/string functions differential-tested against the host where contracts overlap; (2) startup and bounded formatting; (3) aligned bump allocator; (4) free-list allocation, split, insertion, and coalescing; (5) invariant checker and deterministic heap dump; (6) seeded randomized traces replayable from a log.

Run `python3 ftt start lab-04-allocator` and check it with `python3 ftt check lab-04-allocator`.

Acceptance rubric: 25% memory primitives, 30% allocator correctness, 15% corruption and overflow detection, 10% deterministic stress tests, 10% ABI/freestanding build, and 10% invariant explanation. A passing happy-path demo cannot compensate for overlapping live allocations.

## Reverse-engineering lens

Given only an arena dump and a few returned pointers, reconstruct header size, alignment, free-list order, and the operation that caused corruption. Mark facts separately from hypotheses, then use one additional allocation to discriminate between competing models.
