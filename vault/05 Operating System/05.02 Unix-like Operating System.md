# 05.02 Unix-like Operating System

## Why this block exists

The kernel is the policy layer that turns CPU privilege, address translation, timers, and devices into isolated processes and durable interfaces. Building a constrained Unix-like model connects every previous artifact through observable state transitions.

## Lecture

### Mental model

The kernel is an event-driven state machine entered by boot, trap, interrupt, or system call. A process combines an address space, register context, lifecycle state, parent relationship, and descriptor table. Open-file descriptions hold shared offset and status; descriptors are per-process references to them. A scheduler chooses among runnable processes but does not manufacture runnable state.

Implement in layers: boot and trap frame, physical-page allocator, virtual mappings, syscall copy-in/copy-out, descriptors, cooperative tasks, timer preemption, process lifecycle, `fork`, `exec`, `wait`, and `mmap`. Every transition should have one owner and a cleanup path.

### Invariants

- Exactly one running process exists per CPU; every queued process is runnable and appears once.
- Kernel code never directly trusts a user pointer; copy helpers validate the complete range and permissions.
- Process, mapping, page, and open-file reference counts equal actual ownership.
- An exited process cannot run; its parent can reap it once.
- Failed syscalls either leave externally visible state unchanged or document a precise partial effect.

### Safety and limits

Use an emulator and file-backed fixtures, not privileged host operations. Keep the first scheduler single-core and deterministic. The course kernel is not hardened against hostile native code and omits signals, users, SMP, production security, and a complete POSIX surface. Bind any host adapters to localhost.

## Workbook

Trace two processes through `open`, `fork`, interleaved `read`, `close`, child `exit`, parent `wait`, and `exec`. Predict descriptor tables, shared offsets, reference counts, process states, run queue, and reclaimable resources after each event. Then inject a copy-out fault halfway through `read` and apply the documented partial-effect rule.

For a timer interrupt during a syscall, draw the privilege and stack transitions from user registers to trap frame, scheduler, and restored context.

## Problem set

1. Distinguish a descriptor from an open-file description and show how `fork` affects both.
2. Find the leak in an `exec` path that destroys the old address space before the new image is fully validated; design a transactional replacement.
3. Specify process-state transitions for spawn, block, wake, exit, wait, orphaning, and reaping.
4. Explain how to validate a user buffer that crosses a mapped page into an unmapped page without integer overflow.
5. Construct a scheduler trace showing starvation under one policy and revise the policy while retaining deterministic tests.
6. State reference-count updates for mapping the same physical page twice and then unmapping each view.

## Homework

Build a staged kernel simulator: (1) traps and context records; (2) physical and virtual memory managers; (3) checked user copies and basic I/O syscalls; (4) process/run-queue state machine; (5) file descriptors and blocking; (6) `fork`, atomic `exec`, `wait`, `exit`; (7) `mmap`, protection changes, and invariant audit after every scripted event.

Acceptance rubric: 20% trap/syscall correctness, 20% process lifecycle, 15% memory isolation, 15% descriptor ownership, 10% deterministic scheduling, 10% failure rollback/cleanup, and 10% learner-written scenario tests. Include a trace that explains every resource still live at shutdown.

## Reverse-engineering lens

Analyze an unlabeled kernel event trace containing PIDs, trap causes, descriptor changes, and page faults. Reconstruct which process issued each syscall and identify the first broken invariant behind a later crash rather than stopping at the final symptom.

## Exit criteria

- [ ] Two isolated processes run, block, wake, exit, and are reaped deterministically.
- [ ] User pointers and executable images are fully validated before use or commit.
- [ ] Resource counts return to baseline after scripted workloads.
- [ ] You can reconstruct process and descriptor state from a kernel trace.
