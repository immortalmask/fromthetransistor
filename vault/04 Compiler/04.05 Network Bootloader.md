# 04.05 Network Bootloader

## Why this block exists

A bootloader turns received bytes into executable authority. It is a compact lesson in trust boundaries: transport success does not prove that an image is complete, correctly placed, compatible, or safe to enter.

## Lecture

### Mental model

Define a small framed protocol with sequence number, destination offset, payload length, image identity, and checksum. The receiver is a state machine: idle, receiving metadata, accepting bounded chunks, verifying the complete image, committing it, and transferring control. Keep serial fallback and network transport behind the same image-writer interface.

Validation order matters. Prove header size before reading fields; prove arithmetic before forming a destination range; prove that the range is permitted and does not overlap the running loader; record received coverage; verify the final image hash; only then update the boot target and jump through a documented ABI handoff.

### Invariants

- No packet can write outside the designated staging region or overwrite the loader.
- Duplicate chunks are idempotent; conflicting duplicates fail.
- Completion requires exactly the declared byte coverage, not merely a final packet.
- The committed image is the same bytes whose integrity value was checked.
- Transfer of control occurs once and only from the verified state.

### Safety and limits

Bind fixture networking to localhost and use disposable emulated RAM. A checksum detects accidental corruption but is not authentication; label it accordingly. Do not expose an unauthenticated boot service on a real interface. The core does not provide secure boot, encryption, rollback protection, or production key handling.

## Workbook

Trace an image split into chunks 0, 1, and 2 through delivery order 1, 0, duplicate 1, corrupted 2, corrected 2. Predict state, coverage bitmap, bytes written, acknowledgment, and whether finalization is allowed after each event. Repeat with an offset whose addition wraps the address width.

Draw the memory map and mark loader text/data, stack, staging image, MMIO, and forbidden gaps. For each region, state whether a download may write, execute, or neither.

## Problem set

1. Construct a packet that passes an `offset < size` check but fails `length <= size - offset`; explain the overflow avoided by the latter.
2. Specify behavior for exact duplicates, conflicting duplicates, missing chunks, wrong image IDs, and packets after commit.
3. Compare whole-image checksum, per-packet checksum, and authenticated signature; state which threat each does not address.
4. Design an ABI handoff record containing entry point, memory size, and device information; define validation of every pointer.
5. Explain why writing directly to the final executable region weakens atomicity compared with staging then committing.
6. Choose retransmission and timeout behavior that remains deterministically testable without relying on wall-clock sleeps.

## Homework

Implement: (1) strict header decoder; (2) bounded staging-memory writer; (3) chunk coverage and duplicate handling; (4) image verification; (5) atomic commit and a simulated jump callback; (6) serial and localhost datagram adapters; (7) replayable traces for loss, reordering, corruption, and timeout.

Acceptance rubric: 30% bounds and state-machine correctness, 20% reordering/duplicate behavior, 15% integrity and commit semantics, 15% error recovery/fallback, 10% deterministic fault tests, and 10% handoff documentation. No test should jump to native downloaded bytes.

## Reverse-engineering lens

Recover the protocol from an unlabeled packet transcript and RAM snapshots. Infer header boundaries, sequence semantics, and commit point, then identify the earliest packet proving why a failed boot occurred.

## Exit criteria

- [ ] You can justify validation order from raw packet to control transfer.
- [ ] All address arithmetic is checked before memory access.
- [ ] Loss, duplication, reordering, and corruption have deterministic outcomes.
- [ ] The model never executes unverified native input.
