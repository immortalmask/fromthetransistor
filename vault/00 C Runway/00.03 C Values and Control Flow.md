# C Values and Control Flow

## Why this block exists

Bootloaders, kernels, drivers, and parsers spend most of their time interpreting integers and making control decisions. C's compact syntax hides conversions, overflow, side effects, and undefined cases. You need to predict those decisions before they control a memory write or privilege transition.

## Lecture

A C declaration gives a value a type, not merely a storage size. The type determines representable range, conversion rules, and many operator results. Use ordinary `int` for local counts when its range is adequate; use `<stdint.h>` fixed-width types at encoded boundaries such as instruction fields, registers, sectors, and packet headers.

Key rules for this course:

- Unsigned arithmetic wraps modulo a power of two, but wrapped size arithmetic is usually still a bug.
- Signed overflow is undefined behavior; do not use it as a carry mechanism.
- Integer promotions occur before many operations on smaller types.
- Division by zero and invalid shift counts are not recoverable arithmetic results.
- A condition is true when its value compares nonzero, but explicit comparisons often expose intent.
- Function parameters are passed by value. A pointer value may allow mutation of another object, but the pointer itself was still copied.

Prefer validation at boundaries and invariants internally. For a range `[base, base + length)`, check addition before computing the end. For a cursor, establish `cursor <= limit` and check remaining bytes before advancing.

Keep control flow boring. Early error returns can make preconditions obvious; a single cleanup path can make ownership obvious. Dense expressions with multiple side effects are difficult to trace and may rely on evaluation-order assumptions.

## Workbook

Without running code, complete a value/range table for `int8_t`, `uint8_t`, `int32_t`, `uint32_t`, and `size_t` on your host. Mark which widths are language-guaranteed and which depend on the platform.

Trace these expressions by writing operand types, promoted types, result, and whether behavior is defined:

```c
uint8_t a = 250;
uint8_t b = 10;
int x = a + b;
uint32_t mask = 1u << 31;
int32_t n = -1;
bool p = n < sizeof(int);
```

Then compile a tiny probe and compare. The signed/unsigned comparison deserves special attention: explain the conversion rather than memorizing the result.

Trace this input loop for `"10x"`, `"-1"`, the maximum accepted value, and one above maximum:

```text
read text -> parse -> inspect end pointer -> inspect range -> convert -> act
```

At which step should each input be rejected? What state must remain unchanged after rejection?

## Problem set

1. Write a condition that checks whether `offset + count` fits within `size` without overflowing the addition.
2. Explain the difference between `i++` as a statement and embedding it twice in a larger expression.
3. Why is `char` a poor choice for a raw packet octet when signedness matters?
4. A loop uses `for (size_t i = n - 1; i >= 0; --i)`. Explain the failure and give two safe forms.
5. Classify division by zero, signed overflow, unsigned wrap, and narrowing conversion as defined, undefined, or conversion-dependent behavior.

## Homework

Implement a command-line field decoder in C. Input is a hexadecimal 32-bit word plus a field specification `lsb:width`. Output the unsigned field value and its sign-extended interpretation.

Stage it:

1. Parse without accepting trailing characters.
2. Reject `lsb >= 32`, width zero, width above 32, or `lsb + width > 32` without overflow.
3. Avoid shifting by the type width.
4. Separate parsing, extraction, sign extension, and printing into functions.
5. Test width 1, width 31, width 32, top-bit set/unset, and every rejected boundary.

The artifact is the warning-clean decoder, test table, and an explanation of the width-32 special case.

## Reverse-engineering lens

Decompilers guess types from operations and uses. Challenge those guesses. A register containing `0xffffffff` could represent `-1`, a mask, an address, or four bytes. Follow comparisons, extensions, and shifts to infer intent, and record where C type rules could produce a misleading high-level reconstruction.
