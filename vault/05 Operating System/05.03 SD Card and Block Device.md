# 05.03 SD Card and Block Device

## Why this block exists

Filesystems need a smaller, more reliable contract than “an SD card.” A block-device layer isolates sector addressing, partial failure, latency, and media state so filesystem logic can be tested without real hardware.

## Lecture

### Mental model

A block device exports fixed-size numbered sectors and explicit read/write completion. Translate logical block address to byte range only after checking capacity and multiplication overflow. A file-backed model provides deterministic media; an SD adapter adds command sequencing, card state, timeout, and integrity errors behind the same interface.

Represent each request with operation, starting LBA, count, buffer length, completion status, and transferred count. Decide whether the core API is all-or-error or permits partial completion, then make every caller honor that decision.

### Invariants

- `LBA + count` stays within capacity without wrapping.
- Buffer length exactly covers the requested sector count.
- Completed writes become visible according to one documented ordering rule.
- Read failure never exposes uninitialized bytes as valid data.
- Fault injection is reproducible by seed and request ordinal.

### Safety and limits

Use disposable image copies and never point tests at a real disk device. Cap image size and request count. The core models sector I/O, not flash wear leveling, full SD electrical timing, hot-plug safety, or power-loss atomicity beyond explicitly injected cases.

## Workbook

For a 16-sector image, trace reads and writes at LBA 0, 15, 16, and `(MAX / sector_size) + 1`. Predict byte offsets, bounds outcomes, transferred counts, and resulting bytes. Then inject failure before, during, and after a two-sector write under the chosen completion contract.

Draw the simplified SD initialization state sequence and predict the response to a data command issued before ready state.

## Problem set

1. Write an overflow-safe condition for a request covering `count` sectors from `lba` in a device of `capacity` sectors.
2. Compare all-or-error and partial-completion APIs; show a filesystem bug caused by assuming the wrong one.
3. Specify deterministic faults for timeout, integrity failure, write protection, removed media, and short backing file.
4. Explain why flushing host file buffers is not a complete model of physical power-loss durability.
5. Design a cache key and invalidation rule if a one-sector read cache is added.
6. Given a request trace, locate the first operation that could have produced a corrupted filesystem sector.

## Homework

Implement: (1) bounded file-backed sectors; (2) request/result types; (3) read/write and flush behavior; (4) deterministic failure injection; (5) a simplified SD command/state adapter; (6) trace and replay tool. Test first/last sector, zero count, overflow, readonly media, short image, and failure recovery.

Acceptance rubric: 30% bounds/data correctness, 20% explicit completion semantics, 15% deterministic faults, 15% SD state handling, 10% replayable traces, and 10% tests. All destructive tests must use temporary image copies.

## Reverse-engineering lens

Given a damaged image and block trace, determine which sector changed, which request owned that change, and whether the fault is in address calculation, write ordering, or higher-level filesystem metadata.

## Exit criteria

- [ ] Every request is bounds-checked without overflow.
- [ ] Failure and partial-completion semantics are documented and tested.
- [ ] Tests cannot modify a real host block device.
- [ ] A recorded trace can deterministically reproduce an injected media failure.
