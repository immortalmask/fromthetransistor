# Functions ABI and Object Files

## Why this block exists

The assembler, compiler, linker, bootloader, and kernel must agree on details the C source does not show: where arguments live, who preserves a register, how the stack is aligned, and how unresolved names become addresses. An ABI disagreement can execute plausible instructions before failing far from the call.

## Lecture

Compilation translates each C source file into an object. Assembly encodes instructions. Linking combines objects, resolves symbols, lays out sections, and applies relocations. Loading places executable segments into memory and establishes entry state. Keeping these phases separate makes errors localizable.

A calling convention assigns locations to arguments and returns, caller- and callee-saved registers, stack alignment, and frame behavior. The course's small FTT-16 machine later needs an explicit teaching convention; do not infer one opportunistically in each compiler function.

At a call boundary, draw:

```text
caller live state -> argument placement -> transfer -> callee frame
-> result placement -> restored state -> caller continuation
```

An object symbol may be defined locally, exported, or undefined pending another object. A relocation records where and how the linker must use a final symbol value. It is a typed computation with range rules, not a text replacement.

Host disassembly is an investigation aid. Compiler optimization may remove frames, inline calls, or keep values in registers, so start with an unoptimized debug build, then compare an optimized one deliberately.

## Workbook

Create a two-source C program where one function calls another with three integer arguments. Build object files separately. Inspect symbols and disassembly with available host tools. Record:

- where the call appears;
- which symbol is unresolved before linking;
- where argument values appear under your host ABI;
- which addresses become concrete after linking.

Draw a hypothetical small-machine stack frame with return location, saved state, two local values, and alignment padding. Trace stack pointer values through entry and return. Your exact convention may be hypothetical, but it must be internally consistent.

Given a relocation at address `P` for symbol `S`, compare absolute `S + A` with PC-relative `S + A - P`. Use concrete values and identify which result must fit the encoded field.

## Problem set

1. Distinguish declaration, definition, symbol, section, segment, relocation, and entry point.
2. Why can individually valid object files fail at link time?
3. A callee changes a callee-saved register and does not restore it. Why may the crash occur only after return?
4. An instruction has an 8-bit signed PC-relative displacement. Determine whether a target 140 bytes away is encodable when instructions are 2 bytes and the displacement unit is words.
5. Explain why the stack must obey alignment even when byte-addressed memory could physically hold an unaligned value.

## Homework

Produce an annotated object-file report for a warning-clean multi-file C program:

1. Record compile-only and final link commands separately.
2. List sections and classify code, initialized data, zero-initialized data, and read-only data.
3. List defined and undefined symbols before linking.
4. Annotate one call from argument setup through return using disassembly/debugger evidence.
5. Change one symbol name to produce a linker failure; capture and explain it.
6. Compare debug and optimized disassembly without assuming source statements map one-to-one to instructions.

The artifact is the report plus reproducible source/build commands, not the host executable alone.

## Reverse-engineering lens

Function boundaries and prototypes are hypotheses. Call sites reveal argument setup; return uses reveal result width; prologues hint at saved state and local storage; relocation and symbol tables provide stronger names when not stripped. When artifacts disagree with a decompiler, prefer concrete instruction and ABI evidence.
