# 04.04 Ethernet Controller

## Why this block exists

The processor needs a controlled way to exchange packet bytes with the outside model. A software-first Ethernet controller teaches the same ownership, queueing, byte-order, and interrupt issues as hardware without making a real PHY a prerequisite.

## Lecture

### Mental model

Separate the wire-format frame from the device interface. Frames contain destination and source addresses, type/length, payload, padding, and an integrity check. The controller exposes MMIO configuration and producer/consumer descriptor rings. Each descriptor names a bounded buffer and has one owner at a time: software prepares TX then hands it to the device; the device fills RX then hands it to software.

Use a deterministic backend that accepts and emits fixture frames and optionally writes PCAP. Model completion and interrupt status explicitly. Ring indices wrap modulo capacity, while a generation bit or count distinguishes full from empty.

### Invariants

- Software and controller never own the same descriptor simultaneously.
- Buffer address plus length stays within configured RAM and the maximum frame size.
- Consumer order follows producer order modulo the documented ring policy.
- Status bits are acknowledged without losing an event that arrives concurrently.
- Frame fields use network byte order regardless of host byte order.

### Safety and limits

Do not require raw sockets, root privileges, or Internet access. Reject oversize, truncated, and bad-integrity frames before exposing payload. The core is a behavioral MAC model, not a timing-accurate PHY and not an implementation of IP or TCP.

## Workbook

For a four-entry ring, predict producer index, consumer index, owner bit, interrupt status, and full/empty state across enqueue A, B, C, consume A, enqueue D, E, consume B–E. Then inject one completion between a status read and acknowledge and explain how the register semantics avoid losing it.

Decode a supplied frame by hand. Mark byte offsets and endian conversion for both MAC addresses, EtherType, payload length, padding, and checksum evidence before using the parser.

## Problem set

1. Give two representations that disambiguate full from empty when producer equals consumer; compare their update cost.
2. Find the bounds error in validating `address < RAM_SIZE` without also validating `address + length`.
3. Specify device and software actions for TX success, TX overflow, RX bad checksum, and RX ring full.
4. Explain why packed C bitfields are a poor interchange format for descriptors shared with modeled hardware.
5. Calculate the bytes of an Ethernet header for given addresses and EtherType, showing byte order.
6. Design a deterministic test for an interrupt arriving during acknowledgment.

## Homework

Build: (1) frame encode/decode with golden vectors; (2) bounded RAM buffer access; (3) TX descriptor lifecycle; (4) RX lifecycle and filtering; (5) sticky status and interrupt acknowledgment; (6) fixture/PCAP backend and fault injection. Emit a concise trace containing cycle, register action, descriptor, old owner, new owner, and result.

Acceptance rubric: 25% frame correctness, 30% ring/ownership behavior including wraparound, 15% bounds and malformed-input rejection, 10% interrupt semantics, 10% deterministic traces, and 10% learner tests. Keep real-network access outside the acceptance path.

## Reverse-engineering lens

Given an MMIO trace with register names removed, infer which accesses are producer updates, completion reads, and acknowledgments. Correlate descriptor changes with a packet capture and locate the first event that explains a dropped frame.

## Exit criteria

- [ ] You can trace descriptor ownership through ring wraparound.
- [ ] Frame parsing is bounded and endian-explicit.
- [ ] Malformed descriptors cannot access memory outside configured RAM.
- [ ] A captured frame can be linked back to the MMIO operation that produced it.
