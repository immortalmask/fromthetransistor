# 05.04 FAT Filesystem

## Why this block exists

FAT is simple enough to inspect by hand yet rich enough to teach on-disk geometry, allocation chains, directory records, corruption handling, and update ordering. It turns the block API into persistent named data.

## Lecture

### Mental model

Mounting derives a geometry from the boot sector: reserved area, FAT copies, root-directory area where applicable, data start, cluster size, and cluster count. A directory entry maps a name and size to a first cluster. The FAT maps each cluster to another cluster, end-of-chain, free, bad, or reserved. File offset selects a chain position and an offset within that cluster.

Parse fields explicitly as little-endian bytes; do not cast disk data to packed C structs. Validate derived regions before trusting them. Traverse with both a maximum-step bound and cycle detection. For updates, stage data and allocation changes so an interruption has a documented, inspectable result.

### Invariants

- Every referenced cluster lies in the data-cluster range.
- Chain traversal terminates at end-of-chain within the cluster-count bound and never cycles.
- File size never causes reads beyond its validated chain or image.
- Allocated clusters have at most one owner in the supported model.
- Mirrored FAT copies agree after a successful committed update.

### Safety and limits

Operate on disposable images, never mounted host filesystems. Enforce maximum image, directory, and chain sizes. The core supports one explicitly documented FAT variant and short names before optional long-file-name work; it is not a repair tool for valuable media.

## Workbook

Given a boot sector and FAT bytes, calculate every region boundary and decode a three-cluster file by hand. Predict the exact sectors read for offsets 0, cluster-size minus 1, cluster-size, and end-of-file. Modify one FAT entry to form a cycle and state which check detects it.

Plan creation of a two-cluster file. List data writes, directory update, and FAT-copy writes in order, then describe observable states if interruption occurs after each operation.

## Problem set

1. Derive data-start sector and cluster count from supplied boot-sector fields, checking every multiplication and addition.
2. Decode packed FAT entries from raw bytes for free, next-cluster, bad, and end-of-chain values.
3. Diagnose a file whose size requires three clusters but whose chain contains two; define safe read behavior.
4. Compare a visited bitmap with tortoise-and-hare cycle detection for memory use and forensic diagnostics.
5. Design allocation rollback when the second of three requested clusters cannot be written.
6. Explain why trusting a null-terminated on-disk name can read beyond a directory record.

## Homework

Implement: (1) bounded boot-sector parser and geometry; (2) FAT-entry decoder and guarded chain walk; (3) directory listing and file read; (4) cluster allocation and file creation; (5) truncate/delete with explicit update ordering; (6) consistency checker for cycles, cross-links, lost clusters, and size mismatch. Preserve golden malformed images.

Run `python3 ftt start lab-05-fat`, then use `python3 ftt check lab-05-fat`.

Acceptance rubric: 20% geometry, 25% read/traversal, 20% safe updates, 15% corruption rejection, 10% host-tool interoperability on disposable images, and 10% forensic explanations. No unchecked on-disk offset may become a host memory access.

## Reverse-engineering lens

Recover files from a supplied image without mounting it. Correlate boot geometry, directory entries, cluster chains, and slack bytes; distinguish active data, deleted metadata, and unsupported speculation in your report.
