# 06.04 Text Web Browser

## Why this block exists

The browser is the software-first capstone: it composes sockets, partial I/O, allocation, parsing, data structures, layout, and terminal rendering into a user-visible result without pretending to implement the modern Web platform.

## Lecture

### Mental model

Keep a strict pipeline: parse URL, connect, serialize request, incrementally parse response framing, decode body bytes, tokenize the supported HTML subset, build or stream a bounded document model, compute block/inline layout for a terminal width, then render plain text or ANSI. Each stage owns one representation and reports errors with byte or node location.

HTTP is a byte protocol, not a line-oriented socket guarantee. Reads may split anywhere. Determine body framing from the selected course subset before consuming it. HTML tokenization also spans arbitrary chunks and requires explicit states for text, tags, attributes, entities, and malformed input. Layout should be deterministic for a given document, width, and ANSI mode.

### Invariants

- Network chunk boundaries never affect the parsed response or rendered document.
- Declared/body limits are enforced before allocation or append.
- Tokenizer input is consumed monotonically; malformed input cannot cause an infinite loop.
- Link indexes map to normalized, bounded targets displayed to the learner.
- Plain-text mode contains no terminal-control bytes from the document.

### Safety and limits

Core fetching is restricted to the provided localhost fixture server. Strip or escape control characters before terminal output. Cap headers, body, tokens, nodes, nesting, redirects, line width, and history. Explicitly exclude TLS, JavaScript, CSS, images, cookies, authentication, arbitrary downloads, and production Internet compatibility.

## Workbook

Take one response whose status line, headers, chunk-size fields, tags, entity, and multibyte text are split across awkward read boundaries. Before running code, predict and record HTTP parser state, remaining body count, HTML tokenizer state, open elements, current line width, and rendered output after each chunk. Re-run with different boundaries and demand identical output.

At widths 20 and 40, lay out a heading, paragraph, emphasized word, long unbreakable token, and three links by hand. State the wrapping rule for each.

## Problem set

1. Parse responses using connection-close, fixed length, and the course's chunked subset; identify conflicting or unsupported framing safely.
2. Design incremental states for a tag and quoted attribute split across five reads.
3. Show an input that makes a parser loop without consuming bytes; add and test a progress invariant.
4. Normalize a relative link against a base path while rejecting unsupported schemes and path growth beyond limits.
5. Specify terminal escaping for ESC, control bytes, tabs, newlines, and malformed encoded text.
6. Derive golden wrapping output for nested blocks at two terminal widths and distinguish parser errors from layout errors.

## Homework

Implement: (1) URL/request model; (2) bounded incremental HTTP response parser; (3) HTML tokenizer and constrained tree/document representation; (4) text extraction and entities; (5) width-aware layout and plain/ANSI renderers; (6) link selection and history; (7) localhost fixture server with fragmented, malformed, oversized, and redirect cases; (8) end-to-end deterministic snapshots.

Run `python3 ftt start lab-06-browser`, then check with `python3 ftt check lab-06-browser`.

Acceptance rubric: 20% HTTP streaming/framing, 20% HTML streaming and malformed input, 20% deterministic layout/rendering, 15% bounds and terminal safety, 10% navigation/history, 10% end-to-end fixtures, and 5% architecture explanation. Passing one well-formed page is not sufficient.

## Reverse-engineering lens

Given only a wrong terminal snapshot, fragmented transport log, and token trace, locate the first layer whose output diverges. Classify the defect as framing, tokenization, tree construction, layout, escaping, or rendering and support the claim with one minimal fixture.

## Exit criteria

- [ ] Rendered output is invariant under all tested network chunk boundaries.
- [ ] Malformed and oversized responses fail safely with bounded resources.
- [ ] The browser renders and navigates the documented HTML subset from localhost.
- [ ] You can localize an end-to-end defect to one intermediate representation.
