# Blinking an LED

## Why this block exists

A blinking output is the traditional first hardware program because it combines clocked state, reset, width, timing arithmetic, and an observable pin. The visual blink is only a demo; the real artifact is a transition model whose period and reset behavior are asserted exactly.

## Lecture

Let the input clock be `F` edges per second. If an output toggles every `N` edges, its full on/off period is `2N/F` seconds and frequency is `F/(2N)`. Decide whether the counter commits a toggle after seeing values `0..N-1` or `0..N`; that one comparison changes the period.

A robust divider contract states:

- reset value for counter and LED;
- whether reset is synchronous;
- exact edge that toggles output;
- width and legal range of the terminal count;
- what happens if the parameter is zero or cannot fit.

For simulation, use a tiny terminal count so wrap/toggle behavior appears within a few cycles. Do not simulate fifty million cycles merely to create a human one-second blink.

The portable core representation is a state struct and `step(reset)` function in C or Python. It makes old state, next state, and trace output explicit. If Verilator is installed, express the same contract in synthesizable SystemVerilog and compare traces. The model and RTL must agree on observation phase.

Use [[realhw Branch Study]] as a historical contrast. Its partial board-specific
LED flow supplies real constraints and open questions; your job is to recover
the missing reset, period, tool-version, pin, timing, and observation contracts
before treating a visible blink as evidence.

## Workbook

A divider starts with `count=0`, `led=0`, and toggles when old `count == 2`; otherwise it increments. Trace six rising edges:

| edge | reset | old count | compare | next count | old LED | next LED |
|---:|---:|---:|---:|---:|---:|---:|

Now derive the full LED period in clock edges. Change the rule to “increment first, then compare to 2” and locate the first divergent edge.

For a 50 MHz clock and a requested 1 Hz LED square wave, calculate the half-period terminal edge count and the minimum unsigned counter width. Then choose a simulation parameter that demonstrates the same rule in fewer than 16 edges.

Predict reset behavior when reset is high for two consecutive edges, then deasserted between edges. State what a synchronous model may observe.

## Problem set

1. A counter toggles at values 0 through `N` inclusive. Derive its half-period.
2. Why is “the LED looked right” insufficient evidence for reset priority?
3. Find the minimum width for counts 0 through 24,999,999.
4. Explain the difference between assigning `led = ~led` in a software step and scheduling a nonblocking RTL update at an edge.
5. A parameter calculation uses a 32-bit signed multiplication and overflows. Give a checked formulation or constrain the inputs.

## Homework

Build a resettable LED divider with a deterministic trace:

1. Write a state/next-state contract before code.
2. Implement a portable step model with a configurable terminal count.
3. Assert initial state, repeated reset, first toggle, second toggle, and at least one full period.
4. Introduce an off-by-one comparison deliberately; capture expected and actual traces through the first divergence; repair it.
5. If Verilator is available, implement the equivalent SystemVerilog module and compare its edge trace with the reference model.

The artifact is source, period derivation, tests, and a short evidence trace. It must finish quickly and require no physical LED.

## Reverse-engineering lens

Given only an output transition trace and a known clock, infer candidate divider periods. Then use reset and one internal counter observation to distinguish terminal-count conventions. This is the same strategy used to infer timer peripherals: observed frequencies narrow hypotheses, while boundary traces establish the exact rule.

## Exit criteria

- [ ] You derive full period from the declared toggle rule.
- [ ] Reset, first toggle, and wrap are covered by exact-cycle tests.
- [ ] The artifact runs with a small simulation parameter.
- [ ] One captured trace shows the first off-by-one divergence and repair.
- [ ] If RTL is present, its trace matches the portable reference at every observed edge.
