# Bytes Bits and Representation

## Why this block exists

Every later interface is encoded: instructions, MMIO registers, ELF records, page-table entries, filesystem metadata, Ethernet frames, and protocol text. A value and its byte representation are related but not interchangeable. This lab makes the relationship explicit and testable.

## Lecture

A byte sequence has positions and bit patterns; it does not carry an intrinsic integer type, alignment, signedness, or byte order. Decoding supplies those rules. For little-endian unsigned bytes `b[0..3]`:

```text
value = b0 | b1<<8 | b2<<16 | b3<<24
```

Promote bytes to a sufficiently wide unsigned type before shifting. Never decode untrusted bytes by casting a byte pointer to a wider integer pointer: alignment, object representation, aliasing, and host byte order can all make that invalid or nonportable.

A bit-field contract needs four facts: container width, least-significant bit index, field width, and interpretation. A mask selects; a shift positions. Validate counts before shifting. Keep sign extension a named operation so its preconditions are reviewable.

Hex is notation, not a different stored value. Group hex by the boundary you are investigating: bytes for memory, instruction fields for decode, or nibbles for manual conversion.

The artifact is the utility plus a trace demonstrating one multi-byte decode and one signed-field interpretation.

## Reverse-engineering lens

When a hex dump contains a repeated plausible length every four bytes, do not immediately cast it to a struct. Form hypotheses for byte order, alignment, units, and whether the field includes its own header. Test them against multiple records. A single coincidental value is weak evidence; a decoder that predicts later offsets is stronger evidence.
