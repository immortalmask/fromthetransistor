# 06.02 Telnet Server

## Why this block exists

A network stack becomes useful when multiple independent clients can drive real processes. A deliberately constrained terminal server tests accept loops, process isolation, descriptor inheritance, partial I/O, and cleanup under disconnect.

## Lecture

### Mental model

A listening socket creates connected sockets; it does not itself carry a session. The server repeatedly accepts, creates one child or session task, connects that session's input/output to the socket, and releases the parent's copy. Session state—line editing, terminal negotiation subset, shell process, and resources—must never be shared accidentally between clients.

Treat Telnet control bytes as a small streaming parser rather than assuming all input is text or arrives in lines. If the course implementation chooses a plain-terminal subset, explicitly refuse unsupported negotiations. Backpressure means writes may complete partially and a slow client must not stall unrelated sessions.

### Invariants

- Each connected socket has exactly one owning session and one cleanup path.
- Parent and child close descriptor copies they do not own after handoff.
- Bytes consumed as protocol negotiation never reach the shell as commands.
- Disconnect eventually terminates or detaches the session and reaps its process.
- One client's input, output, cwd, and environment cannot enter another session.

### Safety and limits

Bind only to loopback inside the simulator. Telnet provides no encryption or authentication and must be labeled intentionally insecure. Do not implement an unrestricted host bind shell. Bound clients, line length, output buffering, idle simulated time, and child lifetime.

## Workbook

Trace two clients connecting between parent `accept` calls. Draw every socket/descriptor copy in listener, parent, and session child through login banner, one command, partial output, abrupt disconnect, child exit, and reap. Predict the leak and EOF behavior if the parent forgets one close.

Feed a Telnet negotiation sequence split at every possible byte boundary. Record parser state and emitted application bytes before running the implementation.

## Problem set

1. Draw descriptor ownership for listener, two accepted connections, and two children immediately after both forks and required closes.
2. Explain why one blocking client write can starve another in a naïve single-loop server; propose a course-appropriate remedy.
3. Specify parser behavior for an escaped control byte, unsupported option, truncated command, CR-LF, and CR-NUL.
4. Design cleanup for orderly close, reset, idle timeout, shell exit, and server shutdown.
5. Show how shared current-directory state can leak between sessions if sessions are threads rather than isolated processes.
6. Create a deterministic test proving that output from client A never appears for client B under interleaved scheduling.

## Homework

Build: (1) loopback listener and bounded accept loop; (2) streaming negotiation/line parser; (3) per-session shell process and descriptor handoff; (4) short read/write handling and output backpressure; (5) disconnect and child reaping; (6) two-client transcript harness with forced interleavings and resource accounting.

Acceptance rubric: 25% descriptor/process ownership, 20% stream parsing, 20% concurrent isolation, 15% partial I/O/backpressure, 10% disconnect cleanup, and 10% deterministic tests. The test runner must verify no listeners, children, or descriptors remain after shutdown.

## Reverse-engineering lens

Given interleaved server logs without client labels, reconstruct session membership from socket tuples, PIDs, and byte streams. Identify the first ownership mistake that explains a cross-session leak or hung EOF.

## Exit criteria

- [ ] Two loopback clients receive isolated, concurrent sessions.
- [ ] Negotiation bytes and partial lines are parsed across arbitrary packet boundaries.
- [ ] All disconnect paths close descriptors and reap children.
- [ ] No core command can escape into a host shell or non-loopback interface.
