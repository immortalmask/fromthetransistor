# 05.05 Init Shell and Utilities

## Why this block exists

A kernel becomes a usable system only when ordinary processes can compose its interfaces. `init`, a small shell, and focused utilities expose process, descriptor, path, and filesystem bugs through reproducible user-visible behavior.

## Lecture

### Mental model

`init` is the first user process and the final reaper for orphaned children. The shell is a parser plus process coordinator: read a bounded line, tokenize according to a deliberately small grammar, identify built-ins that must change shell state, otherwise create a child, arrange descriptors, execute, and wait or track background state. Utilities should be thin clients of syscalls, not alternate kernel implementations.

Keep command parsing separate from execution. Represent a command as argv plus redirections and sequencing metadata. Preserve exit status and diagnostics across every layer. Implement robust I/O loops because read and write may complete partially.

### Invariants

- Every spawned child is eventually waited for or intentionally adopted and reaped by `init`.
- Descriptor ownership after fork/exec is explicit; unused pipe ends close in every process.
- Argument vectors and input lines are bounded and correctly terminated.
- Utilities report failure through nonzero status and do not claim success after partial work.
- File replacement uses a documented temporary-write/rename or equivalent policy where supported.

### Safety and limits

Run against the course kernel and disposable filesystem images. The shell intentionally omits arbitrary host command execution, job-control signals, command substitution, globbing, and full POSIX quoting. A `download` utility connects only to the course fixture service on localhost.

## Workbook

Trace `cat input | download output` through parse tree, process creation, descriptor tables, pipe ends, blocking points, exit statuses, and reaping. Predict the hang caused by one extra write-end reference. Then trace `cd /tmp`: explain why implementing it only in a child cannot change the parent shell.

For a sequence that creates, lists, reads, and removes a file, predict every syscall and expected filesystem change before running the transcript.

## Problem set

1. Tokenize five command lines containing spaces, empty arguments, quotes, redirection, and a malformed quote under the documented grammar.
2. Draw descriptor tables for a two-command pipeline before and after each close; identify the minimum leak that prevents EOF.
3. Explain which commands must be shell built-ins and which can be separate executables.
4. Design `cat` and `download` loops that handle short reads, short writes, interruption, and out-of-space errors.
5. Specify `init` behavior for an orphaned child, a repeatedly crashing service, and shutdown.
6. Given a transcript with status codes removed, infer the earliest failing syscall from later observations.

## Homework

Stage userland as: (1) `init` launch/reap loop; (2) bounded tokenizer and command representation; (3) foreground spawn/exec/wait; (4) built-ins and redirection; (5) `cat`, `ls`, `rm`, and a localhost-only `download`; (6) pipelines if kernel support exists; (7) transcript harness that starts from a fresh image and verifies persistence after reboot.

Acceptance rubric: 20% parsing/diagnostics, 20% process and reaping correctness, 20% descriptor/I/O correctness, 15% utilities and persistence, 15% failure-path cleanup, and 10% deterministic transcript tests. No command may escape into the host shell.

## Reverse-engineering lens

Use a syscall trace and disk-image diff to reconstruct an unknown command sequence. Attribute each process, descriptor, and file mutation, then identify whether a failure belongs to parsing, process setup, kernel I/O, or filesystem state.

## Exit criteria

- [ ] Init reaps all children and the shell leaves no descriptor leaks.
- [ ] Parsing and execution are separate, bounded, and deterministic.
- [ ] Utilities handle partial I/O and return meaningful status.
- [ ] A clean scripted session persists expected changes across a simulated reboot.
