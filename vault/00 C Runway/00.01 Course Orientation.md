# Course Orientation

## Why this block exists

The final browser is not a pile of independent tutorials. Its characters depend on HTML parsing, which depends on an HTTP byte stream, which depends on TCP, syscalls, a kernel, compiled C, an ABI, processor execution, and device state. A defect at any lower boundary can imitate a defect above it. Before construction, you need a map, a definition of evidence, and a recovery method.

The course is successful when you can locate and explain causality across layers—not when you accumulate source files or pass tests by trial and error.

## Lecture

The cumulative machine has three kinds of contracts:

- **Representation contracts:** byte order, instruction fields, object records, page-table entries, frames, sectors, and protocol text.
- **Transition contracts:** what changes at a clock edge, instruction, trap, syscall, I/O completion, or packet arrival.
- **Ownership contracts:** who may access a register, buffer, page, descriptor, file, or process, and who releases it.

Every module follows the same proof shape: state a prediction, produce an observable trace, create an artifact, attack its boundary cases, and explain its invariant. Tests are evidence that known observations match; they are not evidence that you understand why.

The required path is entirely software-first. Simulated devices are not “fake” substitutes: they define controllable interfaces, allow fault injection, and produce replayable evidence. Physical timing and electronics appear only after the logical contracts work.

Use version-control checkpoints if Git is available, but do not make course completion depend on Git. Never edit solution assets as learner work. Never make an optional component a hidden prerequisite for a required one.

## Workbook

Draw the dependency path for the final terminal character `H` from the browser down to a device or memory operation. Label at least five boundaries. For each boundary, name one observable trace.

Now predict the purpose and likely result of each command before running it:

```sh
python3 ftt doctor
python3 ftt validate
python3 ftt list --section 00
python3 ftt next
python3 ftt show 00.01 --path
```

Record actual output and answer:

1. Which command checks local executable availability?
2. Which command checks the authored dependency graph and files?
3. Does a structural pass prove any learner lab solution correct?
4. What is the earliest unlocked module, and why?

Create a trace vocabulary table with columns `layer`, `unit of progress`, `state`, and `first useful divergence`. Suggested rows: C expression, RTL, CPU, kernel, filesystem, TCP, browser.

## Problem set

1. A UART echo test fails only when the CPU runs it. List three lower-layer traces and the order in which you would inspect them.
2. A browser test passes after a two-second sleep is added. Explain why this is weak evidence and propose a deterministic observation.
3. Distinguish an artifact, an invariant, an oracle, and a fixture using the assembler as your example.
4. A learner changes the CPU, compiler, and test expectation together. Explain what evidence was destroyed and how to recover it.
5. Classify each as representation, transition, or ownership: signed immediate decoding; RX FIFO dequeue-on-read; freeing a process page.

## Homework

Create a learning-log entry outside the authored vault pages containing:

1. Your available weekly hours and a realistic first four-week checkpoint.
2. The exact local commands that establish a clean baseline.
3. A “last trusted boundary” template with expected state, actual state, first divergence, and fixed reproduction input.
4. One deliberate validation failure you can safely create in temporary learner work, followed by its restoration and captured before/after result.
5. A scope contract: core simulation required; dynamic linking and physical hardware optional; no real-network bind shell; no unreviewed hardware power-up.

The artifact is the verified checkout, learning log, and scope contract—not a modification to the course source.

## Reverse-engineering lens

Reverse engineering begins with observations and constraints, not guesses about source. Treat the course machine the same way. If given only a trace, ask which state is visible, what event advances it, and which rules could produce the next row. Your existing instinct to work backward from bytes and behavior is valuable; this course adds forward construction and explicit invariants.
