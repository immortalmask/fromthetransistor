# 04.02 ELF Linker

## Why this block exists

Separate compilation is only useful when independently produced code and data can be placed into one address space without losing their references. The linker is where names become addresses and where many “compiler” bugs are finally exposed.

## Lecture

### Mental model

An object file is a set of byte-bearing sections plus claims about names and unfinished address calculations. Linking has four explicit phases: read and validate objects, merge compatible input sections into output sections, resolve global symbols, then apply relocations using the final layout. Emit a map file so every output byte has a provenance.

Start with a documented teaching object format, then serialize the same concepts as ELF32: ELF header, section table, string tables, symbol table, relocation records, and loadable program headers. Section headers describe linking metadata; program headers describe what a loader maps. They are not interchangeable.

### Invariants

- Every defined global symbol has exactly one winning definition; every required symbol resolves.
- Output section starts satisfy their alignment, and contributions never overlap.
- A relocation writes only within its target section and its value fits the relocation width.
- Symbol values in the map equal the addresses encoded into relocated references.
- Identical ordered inputs produce byte-identical output.

### Safety and limits

Treat all object lengths, counts, offsets, and string indexes as hostile. Check arithmetic before slicing or allocating. Never execute linked output merely because parsing succeeded. The core supports a small fixed relocation set and static executable images; shared objects and dynamic relocations arrive later.

## Workbook

Lay out `.text` contributions of 13 and 9 bytes at alignment 4, followed by `.rodata` of 7 bytes at alignment 8 and `.data` of 5 bytes at alignment 4. Predict every start, padding byte, symbol address, and final size. Then apply by hand an absolute 32-bit relocation and a signed PC-relative relocation using `S`, `A`, and `P`.

Inspect a small host object with `readelf` or `objdump`. Before reading the relocation table, predict which source references cannot be finalized by the compiler. Explain each actual relocation without relying only on its mnemonic.

## Problem set

1. Resolve a table containing local, global, undefined, and weak symbols; identify all errors and winners.
2. Compute `S + A` and `S + A - P` for three relocations, including one overflow, using hexadecimal arithmetic.
3. Given four input sections with different alignments, construct the minimum-padding output layout and prove no regions overlap.
4. Explain why section metadata may be stripped from an executable while program headers remain necessary to load it.
5. Design an error for a relocation whose offset points two bytes before the end of a section but writes four bytes.
6. Compare duplicate-symbol and unresolved-symbol failures: at what phase is each knowable, and what provenance should be printed?

## Homework

Implement: (1) strict object parsing with bounded reads; (2) ordered section merging and alignment; (3) local/global symbol resolution; (4) a small relocation table; (5) deterministic flat-image and ELF32 emission; (6) a human-readable map. Test empty sections, large alignment, undefined/duplicate names, overflow, corrupt offsets, and reordered inputs.

Run `python3 ftt start lab-04-linker`, then verify with `python3 ftt check lab-04-linker`.

Acceptance rubric: 30% correct layout, 25% symbol and relocation correctness, 15% malformed-input rejection, 10% deterministic ELF structure, 10% useful map/diagnostics, and 10% learner tests. External inspection tools must agree with header and segment claims.

## Reverse-engineering lens

Remove source labels from a two-object program and reconstruct its layout from the ELF and map artifacts. Use relocation sites to distinguish data addresses from branch targets, then verify your hypothesis against the original objects.
